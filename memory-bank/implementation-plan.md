# 实施计划（详细 Todo List）

目标：按 `memory-bank/prd.md` 与 `memory-bank/tech-stack.md` 落地正式版 v1.0。  
约束：Rust 1.92 / Cargo 1.92 / edition 2024；优先 Rust 原生跨平台库；网络默认 rustls；避免 sqlite 等额外 C/C++ 依赖。  
输出：一个可在 Windows/macOS/Linux 运行的 iced 0.14 桌面应用，支持连线分析、连线对战（双向同步）、人机对弈、AI vs AI，具备棋谱与局势曲线等学习闭环能力。正式版工程根目录为仓库下的 `restruct/`。

## 2. xq-core：局面、走法、坐标体系与翻转

- [ ] 2.1 定义核心数据模型：Piece/Side/Board/Move/GameRecord

- [ ] 2.2 实现 FEN 解析与生成（中象 FEN）
  - 产物：`fen -> board -> fen` 可互逆（忽略空格与规范化差异时保持等价）。
  - 测试：用 10+ 组 FEN 样例做 round-trip 测试；包含开局、中局、残棋、空格压缩边界；样例规范以 xqbase 标准为权威基线：

```text
https://www.xqbase.com/protocol/cchess_fen.htm
```

- [ ] 2.3 实现局面差分（识别/同步所需）：从两帧 board 推导 Move/异常类型
  - 产物：返回（正常走子 / 少子 / 多子 / 多处变化 / 不可判定）。
  - 工程策略（以 Demo 思路为基线并增强严谨性）：
    - 先实现与 Demo 一致的最小可用 diff 分类（参考 `server/src/chess.rs:115`）
    - 差异数为 2 且存在 from/to 时判定为 Move；差异数为 1 判定为 OneChanged；其余为 Unknown
    - diff 结果只作为“候选事件”，必须经过确认窗口或状态机校验后才能提交
    - OneChanged 与 Unknown 必须具备明确回退路径（等待下一帧确认、进入对齐流程或降级只读）
    - 识别确认与阈值策略参考 Demo 的确认与重置思路（参考 `server/src/worker.rs:79` 与 `server/src/worker.rs:112`）
  - 测试：构造至少 8 组“前后局面”样例，验证 diff 分类正确（含吃子、被吃、走子、识别漏检等），并增加“候选事件未确认不得落盘”的测试。

- [ ] 2.4 实现棋盘翻转与坐标映射（UI 坐标 ↔ 逻辑坐标）
  - 产物：支持“棋盘反转”后仍能正确落子与高亮提示。
  - 约束：连线相关模式下必须与外部应用方向一致，并禁用手动反转；camp 修正与 UI mirror 的基线思路参考 Demo（`server/src/chess.rs:360` 与 `server/src/worker.rs:104`）。
  - 测试：对 20 个随机坐标做双向映射测试：flip(flip(pos)) == pos；并对 10 个走法做 flip 一致性测试；连线模式下验证反转入口被禁用且方向与外部一致。

- [ ] 2.5 实现走法合法性校验（v1.0 仅覆盖棋子移动规则）
  - 产物：覆盖各子走法规则与基本约束（如将帅照面）；不在 v1.0 内处理长将/长捉/重复局面/60 回合等裁定规则。

## 5. xq-link：外部窗口连线与双向同步（输入注入 + 对齐）

- [ ] 5.1 外部窗口选择与连接（窗口枚举、绑定、基本信息展示）
  - 产物：用户可选择目标窗口并开始/停止连线。
  - 测试：启动后列出窗口列表，选择任意窗口能开始截图循环；停止后 CPU 占用显著下降且无后台任务残留。

- [ ] 5.2 对齐机制：外部识别局面 ↔ 本地局面（Aligned/Desynced/Aligning）
  - 产物：任何时候能判断是否同步；不同步可触发对齐流程（以外部为准或以本地为准，按模式策略）。
  - 测试：人为制造 3 类不同步（识别漏子/用户手动改局面/外部瞬移），验证系统能进入 Desynced 并给出可恢复路径。

- [ ] 5.3 输入注入（跨平台）：将本地走法映射为外部窗口点击序列
  - 产物：支持“点击起点格 → 点击终点格”的最小可靠方案；并可配置点击间隔。
  - 测试：在测试棋盘应用（或目标平台）上连续自动执行 20 步合法走子，验证外部窗口实际落子成功率 ≥ 95%（v1.0 初期门槛，可逐步提升）。

- [ ] 5.4 注入后确认：注入完成 → 触发快速识别 → 确认外部已变化
  - 产物：每次自动落子都必须有确认；失败可重试/提示并回退只读。
  - 测试：在外部窗口故意阻止一次点击（例如遮挡），验证系统能检测确认失败并进入重试/回退流程。

- [ ] 5.5 macOS 权限引导（辅助功能/输入监控）
  - 产物：启用双向同步时检查权限状态，未授权给出明确引导与回退（仍可只读分析）。
  - 测试：在未授权环境尝试启用同步，验证：不会崩溃；提示路径清晰；关闭同步后仍可继续连线分析。

## 7. 棋谱与局势曲线（文件化存储）

- [ ] 7.1 对局记录结构：步序列 + 元信息 + 引擎参数快照 + 评估点
  - 产物：每步可记录一次评估（可采样）；支持关键节点标记（v1.0 最小可先存字段不做 UI）；默认存储位置为 OS 推荐的应用数据目录。
  - 测试：完成一局 20 步对局后保存，重新打开后：步序列、局面回放、评估点数量一致。

- [ ] 7.2 导出：FEN（当前局面）与 PGN（整局）
  - 产物：一键复制/导出；PGN 标签包含基础元信息（时间/模式/红黑/结果等）。
  - 测试：导出后用第三方工具（或你们内部解析器）重新导入，验证走子序列可回放到同一终局局面。

- [ ] 7.3 局势图（优势曲线）与步跳转
  - 产物：曲线随步数增长；点击曲线点可跳转到对应步并更新棋盘/分析。
  - 测试：随机点击 30 个步号跳转，验证：棋盘局面与棋谱高亮步一致；不会出现越界或错位。

- [ ] 7.4 快照（Snapshot）保存与恢复（防崩溃丢失）
  - 产物：
    - 术语统一为“快照”
    - 自动保存不采用时间策略；每一步走子后立即保存快照
    - 连线相关模式不产生快照（仅记录棋谱）
    - 每个客户端最多保留 1 份快照，新快照覆盖旧快照
    - 启动时若存在快照，不自动恢复，应弹窗提示“存在未完成对局，是否继续”
    - 选择“是”：恢复快照记录的模式与对局棋盘状态
    - 选择“否”：进入默认启动状态并删除快照
  - 测试：
    - 在非连线模式对弈 10 步后强制结束进程，重启时应出现提示弹窗
    - 选择“是”后恢复的模式与棋盘状态应与快照一致
    - 选择“否”后应进入默认启动状态且快照被删除
    - 在连线模式下进行对局后强退，重启不应出现快照恢复提示

  - 产物：不同策略在 UI 中可解释；切换策略不破坏当前对局状态。
  - 测试：在同一外部窗口分别跑三种策略各 10 步，验证策略行为符合定义且可安全回退只读。
